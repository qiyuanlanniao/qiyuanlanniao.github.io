LeetCode 第 1631 题：[最小体力消耗路径](https://leetcode.cn/problems/path-with-minimum-effort/description/)。
本题是“二分答案”思想的经典应用。它要求我们寻找一个“最小值”，而这个“值”本身又是由路径上的“最大值”决定的。这种“最小化最大值”的结构，强烈地暗示了我们可以将求解问题转化为判定问题，并利用二分查找来高效解决。
{% endcenterquote %}
<!--more-->

### 问题解读

我们得到一个 `rows x columns` 的二维地图，每个格子 `(row, col)` 都有一个高度值。我们需要从左上角 `(0, 0)` 出发，到达右下角 `(rows-1, columns-1)`。

与常规的寻路问题不同，这里的“成本”或“体力消耗”定义得比较特殊：
1.  我们可以在上、下、左、右四个方向移动。
2.  一条路径的**体力消耗**，被定义为这条路径上**所有相邻格子之间高度差绝对值的最大值**。
3.  我们的目标是找到一条路径，使得这个“体力消耗”值**最小**。

举个例子，`heights = [[1,2,2],[3,8,2],[5,3,5]]`。
*   路径 `[1,3,5,3,5]`：相邻高度差为 `|1-3|=2`, `|3-5|=2`, `|5-3|=2`, `|3-5|=2`。这些差值中的最大值是 2，所以这条路径的体力消耗是 2。
*   路径 `[1,2,2,2,5]`：相邻高度差为 `|1-2|=1`, `|2-2|=0`, `|2-2|=0`, `|2-5|=3`。这些差值中的最大值是 3，所以这条路径的体力消耗是 3。

比较这两条路径，第一条更优。我们的任务就是找出所有可能的路径中，体力消耗值最小的那一个。

### 核心思路：将“求解问题”转化为“判定问题”与二分查找

直接去寻找那条最优路径似乎很复杂，因为路径的成本不是累加的，我们无法直接使用像 Dijkstra 这样的标准最短路算法（虽然稍作修改后也可以）。

一个更巧妙的思路是转换问题角度。我们不去直接问“最小体力消耗是多少？”，而是去问一个更容易回答的判定性问题：**“是否存在一条路径，其体力消耗不超过 `k`？”**

这个问题就简单多了。如果一条路径的体力消耗不超过 `k`，那就意味着这条路径上任意两个相邻格子的高度差都不能大于 `k`。这实际上是在问：在图中，如果我们只保留那些边权（高度差）小于等于 `k` 的边，起点和终点是否还是连通的？

这个问题可以用一次 **广度优先搜索（BFS）** 或 **深度优先搜索（DFS）** 来解决。我们可以写一个辅助函数 `canReach(k)`：
*   将网格看作一个图，每个格子是一个节点。
*   从起点 `(0, 0)` 开始进行图遍历（BFS 或 DFS）。
*   在遍历过程中，只有当一个邻居格子与当前格子的高度差绝对值 `abs(h1 - h2)` 不超过 `k` 时，我们才认为可以移动到那个邻居。
*   如果遍历过程中能够到达终点 `(rows-1, columns-1)`，那么 `canReach(k)` 就返回 `True`，否则返回 `False`。

现在我们有了一个判定方法，并且这个判定结果具有非常重要的**单调性**：
*   如果一个最大体力消耗 `k` 是可行的（即存在这样的路径），那么任何大于 `k` 的体力消耗 `k'` 也一定是可行的。因为允许的差值变大了，原来能走的路径现在依然能走。
*   反之，如果 `k` 是不可行的，那么任何小于 `k` 的值也必然不可行。

这种单调性正是应用**二分查找**的完美场景。我们可以对“体力消耗值”这个答案本身进行二分查找，来快速定位到那个“最小可行”的临界点。

这个二分查找的搜索范围是什么呢？
*   **下界 (left)**：最小可能的体力消耗是 0（如果存在一条路径所有格子高度都相同）。
*   **上界 (right)**：最大可能的体力消耗是地图中的最大高度和最小高度之差，一个宽松的上界是 `10^6 - 1`。

因此，我们就在 `[0, 10^6 - 1]` 这个区间内，对体力消耗值进行二分查找。

### 算法详解

1.  **确定二分查找的范围**
    *   `left = 0`：可能的最小体力消耗。
    *   `right = 10**6`：可能的最高体力消耗（或更精确的 `max_height - min_height`）。

2.  **执行二分查找**
    *   当 `left < right` 时，循环继续。
    *   计算中间值 `mid = left + (right - left) // 2` 作为当前的“猜测体力消耗”。
    *   调用辅助函数 `canReach(mid)` 来判定这个猜测值是否可行。
        *   **如果 `canReach(mid)` 返回 `True`**：
            *   说明体力消耗 `mid` 是可行的。但这不一定是最小的，可能还有更小的可行值。
            *   因此，我们记录 `mid` 作为一个潜在的答案，并尝试在更小的范围里搜索，将搜索范围缩小到左半边：`right = mid`。
        *   **如果 `canReach(mid)` 返回 `False`**：
            *   说明体力消耗 `mid` 太小了，无法从起点走到终点。
            *   我们必须允许更大的体力消耗，因此需要将搜索范围缩小到右半边：`left = mid + 1`。

3.  **返回结果**
    *   循环结束后，`left` 和 `right` 将会相遇在那个最小的可行体力消耗值上，这个值就是我们的答案。

### Python 代码实现

```python
import collections
from typing import List

class Solution:
    def minimumEffortPath(self, heights: List[List[int]]) -> int:
        rows, cols = len(heights), len(heights[0])
        
        # 判定函数：检查在 effort_limit 的限制下，是否存在一条从起点到终点的路径
        def can_reach_destination(effort_limit: int) -> bool:
            q = collections.deque([(0, 0)])
            visited = {(0, 0)}
            
            while q:
                r, c = q.popleft()
                if r == rows - 1 and c == cols - 1:
                    return True
                
                # 遍历四个方向
                for dr, dc in [(0, 1), (0, -1), (1, 0), (-1, 0)]:
                    nr, nc = r + dr, c + dc
                    
                    if 0 <= nr < rows and 0 <= nc < cols and (nr, nc) not in visited:
                        # 只有当高度差不超过限制时，才可通行
                        if abs(heights[nr][nc] - heights[r][c]) <= effort_limit:
                            visited.add((nr, nc))
                            q.append((nr, nc))
            return False

        # 对“体力消耗值”进行二分查找
        left, right = 0, 10**6
        
        while left < right:
            mid = left + (right - left) // 2
            if can_reach_destination(mid):
                # mid 可行，尝试更小的值
                right = mid
            else:
                # mid 不可行，必须增大消耗值
                left = mid + 1
        
        return left
```

### 复杂度分析

*   **时间复杂度**: O(M * N * log(C))。
    *   `M` 和 `N` 分别是地图的行数和列数。
    *   `C` 是高度差的可能范围（即 `10^6`），所以 `log(C)` 是二分查找的迭代次数。
    *   在每次迭代中，我们都需要调用 `can_reach_destination` 函数，该函数执行一次 BFS，最坏情况下会访问所有 `M * N` 个格子，时间复杂度为 O(M * N)。
    *   因此，总时间复杂度是这几部分的乘积。
*   **空间复杂度**: O(M * N)。
    *   空间主要消耗在 BFS 的 `visited` 集合和队列 `q` 上。在最坏情况下，它们可能需要存储地图上所有的格子。

### 总结

本题是二分查找思想应用的绝佳范例。它完美地诠释了如何将一个求解“最优值”（最小的最大值）的复杂问题，转化为一个简单的、具有单调性的“可行性判定”问题。一旦识别出这种模式，就可以套用二分查找的框架，将一个巨大的搜索空间通过对数次判定，快速收敛到最终答案，这是一种非常强大且高效的算法思想。
