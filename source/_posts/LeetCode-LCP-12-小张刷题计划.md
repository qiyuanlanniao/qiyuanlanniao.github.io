LeetCode LCP 12. [小张刷题计划](https://leetcode.cn/problems/xiao-zhang-shua-ti-ji-hua/description/)。
直接去计算“最小的 T”似乎无从下手，因为它涉及到如何分组、如何使用求助等复杂决策。如果我们**猜测**一个 T 值（例如，T=3），我们能否判断出，是否存在一种方案，能在 m 天内完成所有题目，并且每天的耗时都不超过 T？

<!--more-->

### 问题解读

小张有 n 道题要刷，计划在 m 天内按顺序完成。完成第 `i` 道题需要 `time[i]` 的时间。为了加速，他每天有一次“场外求助”的机会，可以让一道题的耗时变为 0。

我们的目标是合理安排每天要做的题目以及求助机会，使得 m 天中“做题时间最多的一天”的耗时 T 尽可能小。我们需要求出这个最小的 T 是多少。

举个例子，`time = [1,2,3,3], m = 2`。
*   我们可以这样安排：
    *   **第一天**：做第 0、1、2 题。总耗时是 `1 + 2 + 3 = 6`。但我们有一次求助机会，应该用在耗时最长的题目上，即第 2 题（耗时3）。所以第一天实际耗时 `1 + 2 = 3`。
    *   **第二天**：做第 3 题。耗时是 3。我们可以对这道题使用求助，所以第二天实际耗时 0。
*   这样安排下来，两天的耗时分别是 3 和 0。其中耗时最多的一天是 3。我们可以证明，不存在耗时更少的方案，所以答案是 3。

### 核心思路：将“求解问题”转化为“判定问题”与二分查找

直接去计算“最小的 T”似乎无从下手，因为它涉及到如何分组、如何使用求助等复杂决策。

但是，我们可以换一个角度思考：如果我们**猜测**一个 T 值（例如，T=3），我们能否判断出，是否存在一种方案，能在 m 天内完成所有题目，并且每天的耗时都不超过 T？

这个“判定问题”就变得清晰了。我们可以设计一个辅助函数 `check(T)`：
*   该函数接收一个最大耗时 T 作为参数。
*   它会贪心地尝试在不超过 T 的前提下，每天尽可能多地刷题。
*   如果能在 m 天或更少天内完成所有题目，则说明这个 T 是可行的，函数返回 `True`；否则返回 `False`。

有了这个判定函数，我们发现它具有非常明显的**单调性**：
*   如果耗时上限 `T` **可以**让小张在 m 天内完成计划，那么任何大于 `T` 的 `T'`（更宽松的上限）也**一定**可以完成。
*   反之，如果 `T` **不可以**完成计划，那么任何小于 `T` 的 `T''`（更严格的上限）也**必然**无法完成。

这种单调性是应用**二分查找**的完美信号。我们可以对“耗时 T”这个值进行二分查找，来快速定位到那个“最小可行”的临界点。

这个二分查找的搜索范围是什么呢？
*   **下界 (left)**：0。在最理想的情况下（例如题目数量小于等于天数 m），小张可以每天只做一题并求助，耗时为 0。
*   **上界 (right)**：所有题目的总耗时。这是最坏的情况，即 m=1，一天内必须完成所有题目。

因此，我们就在 `[0, sum(time)]` 这个区间内，对 T 进行二分查找。

### 算法详解

1.  **确定二分查找的范围**
    *   `left = 0`：可能的最小耗时。
    *   `right = sum(time)`：必然可行的最大耗时。

2.  **执行二分查找**
    *   当 `left < right` 时，循环继续。
    *   计算中间值 `mid = left + (right - left) // 2` 作为当前的“猜测耗时上限”。
    *   调用辅助函数 `check(mid)` 来判定这个猜测值是否可行。
        *   `check(mid)` 函数内部，使用贪心策略模拟刷题过程：
            *   初始化天数 `days_needed = 1`，当天已用时间 `current_day_time = 0`，当天遇到的最长耗时 `max_time_this_day = 0`。
            *   遍历每一道题 `t` in `time`：
                *   更新当天遇到的最长耗时 `max_time_this_day = max(max_time_this_day, t)`。
                *   判断如果把当前这道题 `t` 加入今天的计划，在使用了求助后，时间是否会超限。检查条件是 `current_day_time + t - max_time_this_day > mid`。
                    *   **为什么是这个条件？** 因为在当天，我们可以随时决定把求助机会用在已经遇到的、耗时最长的那道题上 (`max_time_this_day`)。所以当天的净耗时就是 `current_day_time + t - max_time_this_day`。
                *   如果该条件成立（即时间超限），说明当前这道题 `t` 无法放入今天。我们必须开启新的一天。
                    *   `days_needed` 加 1。
                    *   新的一天从当前这道题 `t` 开始，所以 `current_day_time = t`，`max_time_this_day = t`。
                *   如果未超限，则将 `t` 加入今天的计划：`current_day_time += t`。
            *   遍历结束后，返回 `days_needed <= m`。
        *   **如果 `check(mid)` 返回 `True`** (即 `mid` 这个上限是可行的)：
            *   说明 `mid` 是一个潜在的答案，但我们想找的是“最小”的 T，所以我们尝试一个更小的值，将搜索范围缩小到左半边（包含 mid）：`right = mid`。
        *   **如果 `check(mid)` 返回 `False`** (即 `mid` 这个上限太严格了)：
            *   说明我们需要放宽上限，将搜索范围缩小到右半边：`left = mid + 1`。

3.  **返回结果**
    *   循环结束后，`left` 和 `right` 相遇，此时的值就是满足条件的最小耗时上限 T。

### Python 代码实现

```python
from typing import List

class Solution:
    def minTime(self, time: List[int], m: int) -> int:
        
        # 判定函数：在每天耗时上限为 T 的情况下，能否在 m 天内完成所有题目
        def check(T: int) -> bool:
            days_needed = 1
            current_day_time = 0
            max_time_this_day = 0
            
            for t in time:
                max_time_this_day = max(max_time_this_day, t)
                # 检查如果把当前题目 t 加入今天的任务，并对今天耗时最长的任务使用求助后，
                # 是否会超过上限 T。
                # 注意：这里不能先累加再判断，因为 max_time_this_day 可能会在 t 加入后才更新。
                # 正确的逻辑是，假设我们把 t 加入今天的任务列表，那么今天的任务总耗时是 current_day_time + t,
                # 其中最耗时的任务是 max(max_time_this_day, t)。求助后净耗时为 current_day_time + t - max(max_time_this_day, t)。
                # 这个逻辑比较绕，我们换一种更清晰的。
                # 
                # 换一种思路：我们先尝试将 t 加入今天
                current_day_time += t
                max_time_this_day = max(max_time_this_day, t)

                # 如果今天总耗时（减去一次求助）超过了 T
                if current_day_time - max_time_this_day > T:
                    # 那么这道题 t 不能放在今天，必须开启新的一天
                    days_needed += 1
                    # 新的一天从当前这道题 t 开始
                    current_day_time = t
                    max_time_this_day = t
            
            return days_needed <= m

        # 二分查找的左右边界
        # 左边界：0，理想情况下每天求助，耗时为0
        # 右边界：所有题目的总时间，此时一定可行（m=1的情况）
        left, right = 0, sum(time)
        
        # 二分查找寻找最小的可行 T
        while left < right:
            mid = left + (right - left) // 2
            if check(mid):
                # 如果 mid 时间可行，说明它可能是答案，
                # 或者还有更小的可行时间，所以我们尝试在左半部分 [left, mid] 继续寻找
                right = mid
            else:
                # 如果 mid 时间不可行，说明时间太少，
                # 必须增加，所以去右半部分 [mid + 1, right] 寻找
                left = mid + 1
        
        # 循环结束时，left 就是满足条件的最小耗时
        return left

```

### 复杂度分析

*   **时间复杂度**: O(N * log(S))。
    *   二分查找的搜索空间大小为 `S = sum(time)`，因此迭代次数为 O(log(S))。
    *   在每次迭代中，我们都需要调用 `check` 函数。`check` 函数需要一次遍历 `time` 数组，时间复杂度为 O(N)，其中 N 是题目的数量。
    *   因此，总时间复杂度是这两者的乘积，即 O(N * log(S))。
*   **空间复杂度**: O(1)。
    *   `check` 函数中只使用了常数个额外变量，因此空间复杂度为 O(1)。

### 总结

本题是“最小化最大值”问题的一个经典应用。它完美地展示了如何通过二分答案，将一个复杂的求解问题转化为一个相对简单的、具有单调性的判定问题。当我们遇到求解“最小的...最大值”或“最大的...最小值”这类问题时，应优先考虑是否能套用二分答案的框架，这往往是通向高效解法的钥匙。
