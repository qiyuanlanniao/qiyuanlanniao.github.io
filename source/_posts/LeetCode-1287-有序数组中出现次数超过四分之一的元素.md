LeetCode 第 1287 题：[有序数组中出现次数超过四分之一的元素](https://leetcode.cn/problems/element-appearing-more-than-25-in-sorted-array/description/)。
这道题的优雅解法，完美诠释了如何利用问题的内在约束（“超过25%”）和数据特性（“有序”），将一个看似需要线性扫描的问题，转化为仅需常数次检查的对数级别算法。

<!--more-->
### 算法优化特性：从 O(n) 到 O(log n) 的飞跃

面对这个问题，最直观的想法是遍历整个数组，使用哈希表或计数器来统计每个数字的出现次数，时间复杂度为 O(n)。但题目给出了一个至关重要的线索：**数组是有序的**。这个特性让我们有机会使用二分查找来进行优化。

然而，对谁进行二分查找呢？我们不可能对每个元素都去计算其出现次数，这样效率会更低。

关键的优化点来自对 **“超过 25%”** 这个条件的深度挖掘。如果一个元素在长度为 `n` 的数组中出现次数超过 `n/4` 次，那么这个元素的分佈必然具有一个非常特殊的性质，这使得我们**不需要检查所有元素**，而只需要检查**几个固定的“候选”位置**即可。

这个思想将遍历所有元素的 O(n) 思路，降维成了检查常数个候选元素，每次检查使用二分查找，最终实现了 O(log n) 的时间复杂度。

### 正确性证明：为何只需检查几个点？

算法的核心是基于一个类似**鸽巢原理**的简单证明：一个长度超过 `n/4` 的连续区块，不可能完全“躲藏”在数组的四个“角落”里，它必然会跨越我们设定的几个关键“检查点”。

下面我们来严谨地证明这一点。

1.  **定义问题**
    *   设数组 `arr` 的长度为 `n`。
    *   我们要找的特殊整数为 `x`，其出现次数为 `k`。
    *   根据题意，`k > n / 4`。由于 `k` 是整数，这等价于 `k >= floor(n / 4) + 1`。

2.  **利用有序性**
    *   因为数组是**非递减**的，所有 `k` 个 `x` 必定是**连续排列**的。它们会形成一个长度为 `k` 的连续区块。

3.  **设置检查点**
    *   我们可以将数组从概念上分为四段。最直观的分割点（即我们的“检查点”）是位于 `n/4`, `n/2`, 和 `3n/4` 位置上的元素。
    *   为了方便，我们考虑下取整的索引：`idx_1 = n // 4`, `idx_2 = n // 2`, `idx_3 = 3 * n // 4`。
    *   我们的论点是：特殊整数 `x` **必然**是 `arr[idx_1]`, `arr[idx_2]`, `arr[idx_3]` 中的一个。

4.  **反证法证明**
    *   假设特殊整数 `x` **不是**这三个检查点中的任何一个。
    *   那么，`x` 所形成的长度为 `k` 的连续区块，必须完整地存在于由这些检查点分割出的四个“间隙”之一中：
        1.  `[0, idx_1 - 1]`
        2.  `[idx_1 + 1, idx_2 - 1]`
        3.  `[idx_2 + 1, idx_3 - 1]`
        4.  `[idx_3 + 1, n - 1]`
    *   现在我们来计算这些“间隙”的最大长度。最长的间隙是第一个，其长度为 `(idx_1 - 1) - 0 + 1 = idx_1 = n // 4`。其他间隙的长度都不会超过 `n // 4`。
    *   我们得到一个结论：任何不包含检查点的连续区间，其长度**最大为 `n // 4`**。
    *   这与我们已知的事实——`x` 的连续区块长度 `k` **严格大于 `n / 4`**——产生了矛盾！
    *   因此，最初的假设（`x` 不是任何一个检查点元素）是错误的。

**结论：** 特殊整数 `x` 必然是 `arr[n//4]`, `arr[n//2]`, 或 `arr[3*n//4]` 中的一个（或多个）。我们只需要将这三个位置的元素作为候选者，验证它们中哪一个的出现次数超过 `n/4` 即可。

### 巧妙的实现

验证一个候选者 `cand` 的次数，最直接的方法是找到它的左右边界：`count = bisect_right(arr, cand) - bisect_left(arr, cand)`。

而下列代码则采用了一种更精炼的“前向检查”技巧，避免了第二次二分查找：

```python
import bisect
from typing import List

class Solution:
    def findSpecialInteger(self, arr: List[int]) -> int:
        n = len(arr)
        # 任何出现次数 > n/4 的元素，其个数至少为 m + 1
        m = n // 4

        # 候选检查点的索引。其原理是：长度 > m 的连续块必然会跨过这些点之一。
        # 这里选择的点集 (m, 2m+1, ...) 也能起到分割作用，确保间隙长度不大于 m
        candidates_indices = (m, m * 2 + 1, m * 3 + 2)
        
        for i in candidates_indices:
            # 数组边界保护
            if i >= n:
                continue
            
            candidate = arr[i]
            
            # 1. 找到该候选者第一次出现的位置
            first_occurrence_idx = bisect.bisect_left(arr, candidate)
            
            # 2. 精髓：从第一次出现的位置向后数 m 个位置
            # 如果目标索引仍在数组界内，并且其值仍然是 candidate
            # 则证明从 first_occurrence_idx 到 first_occurrence_idx + m
            # 这 m + 1 个元素都是 candidate，计数达标。
            target_idx = first_occurrence_idx + m
            if target_idx < n and arr[target_idx] == candidate:
                return candidate
        
        # 基于题目“恰好有一个”答案的保证，在大多数情况下循环会提前返回。
        # 为覆盖所有边界情况（例如数组长度很短），返回 arr[0] 作为保底。
        return arr[0]

```

这段代码的巧妙之处在于：
1.  **高效验证**：通过 `bisect_left` 找到起始点 `j`，然后直接检查 `arr[j + m]` 的值。如果它仍然是我们的候选者，就意味着从 `j` 到 `j+m`（共 `m+1` 个）都是同一个数，直接满足了 `> n/4` 的条件。这比“找左边界再找右边界”的方式更高效。
2.  **候选者选择**：选择 `m`, `2m+1`, `3m+2` 等作为检查点索引，同样可以保证覆盖所有可能性，因为它们之间的间隔小于等于 `m+1`，而目标元素的区块长度大于 `m`。
3.  **确定性**：由于题目保证了答案“恰好有一个”，所以我们测试的几个候选者中必然会有一个满足条件并被返回。

### 复杂度分析

*   **时间复杂度**: O(logn)，其中 n 是 arr 的长度。
*   **空间复杂度**: O(1)。
