LeetCode 3733 | 完成所有送货任务的最少时间

发表于 2025-11-16 分类于 算法题解 ， 二分算法 ， 二分答案 ， 最小化最大值 阅读次数： Valine：

LeetCode 第 3733 题：完成所有送货任务的最少时间。
当直接计算“最少时间”的调度方案非常复杂时，我们可以转换思路，猜测一个“总时间”，然后验证其可行性。这种将求解最值问题转化为判定问题的技巧，正是“二分答案”思想的精髓。

问题解读

我们有两架无人机，分别需要完成 d1 和 d2 次送货任务。每个送货任务都恰好花费一小时。

问题的核心约束有两个：

  1. 时间独占：在任意一个小时内，只能有一架无人机执行送货任务。
  2. 强制充电：无人机 i 每隔 ri 小时需要强制充电，即在第 ri、2*ri、3*ri… 小时，它无法工作。

我们的目标是找到一个最优的调度方案，使得完成所有 d1 + d2 次送货任务所需的总时间最短。

例如，d = [3,1], r = [2,3]。

  • 无人机1需要送货3次，每2小时充一次电（在第2、4、6…小时）。
  • 无人机2需要送货1次，每3小时充一次电（在第3、6、9…小时）。
    一种最优调度是：
  • 第1小时：无人机1送货。
  • 第2小时：无人机1充电，无人机2送货。
  • 第3小时：无人机2充电，无人机1送货。
  • 第4小时：无人机1充电。
  • 第5小时：无人机1送货。
    总共耗时5小时完成所有任务。

核心思路：将“求解问题”转化为“判定问题”与二分查找

直接推导出最优调度策略和最小时间非常困难，因为两架无人机的充电周期相互交错，情况很复杂。

但是，我们可以换一个角度思考：如果我们猜测一个总时间 T，能否判断出，在这 T 小时内，我们是否有可能完成所有的送货任务？

这个“判定问题”就清晰多了。我们可以设计一个 check(T) 函数：

  • 该函数接收一个总时间 T 作为输入。
  • 它需要回答：是否存在一种调度方式，在 T 小时内完成 d1 和 d2 次送货。

这个 check(T) 函数具有非常明显的单调性：

  • 如果时间 T 足够完成所有任务，那么任何比 T 更长的时间 T' 也一定足够。
  • 反之，如果时间 T 不够，那么任何比 T 更短的时间 T'' 也必然不够。

这种单调性是应用二分查找的完美信号。我们可以在一个可能的时间范围内，对“总时间 T”进行二分查找，来高效地定位到那个“恰好足够”的最小时间点。

这个二分查找的搜索范围是什么呢？

  • 下界 (left)：无论如何，至少需要 d1 + d2 个小时才能完成所有送货，所以下界是 d1 + d2。
  • 上界 (right)：我们可以设定一个足够大的数。例如，在最坏的情况下（如充电频率很高），所需时间也不会超过 (d1 + d2) 的几万倍。为保险起见，我们可以设一个较大的值，例如 4 * 10^14。

算法详解：如何实现 check(T)

check(T) 函数是整个算法的核心。我们需要在给定的 T 小时内，判断是否存在一个可行的调度方案。这可以通过一个贪心策略来实现。

首先，我们将 T 小时内的所有时间点（1, 2, …, T）进行分类：

  1. 无人机1独占时段：无人机2在充电，而无人机1空闲。
  2. 无人机2独占时段：无人机1在充电，而无人机2空闲。
  3. 共享时段：两架无人机都空闲。
  4. 阻塞时段：两架无人机都在充电（此时谁也无法工作）。

一个贪心的想法是：让每架无人机优先使用自己的“独占时段”。因为这些时段另一方无法使用，不用白不用。如果独占时段用完后任务还没完成，再从“共享时段”里拿时间。

check(T) 的具体步骤：

  1. 计算各类时段的数量

    • 为了计算在周期 r1 和 r2 的倍数上重叠的部分，我们需要它们的最小公倍数 lcm。lcm(r1, r2) = (r1 * r2) // gcd(r1, r2)。
    • 在 T 小时内，无人机1充电 T // r1 次。
    • 在 T 小时内，无人机2充电 T // r2 次。
    • 在 T 小时内，两者同时充电 T // lcm 次。
    • 因此，我们可以计算出：
      • slots_d1_only (无人机1独占时段数) = (无人机2充电次数) - (两者同时充电次数) = T // r2 - T // lcm
      • slots_d2_only (无人机2独占时段数) = (无人机1充电次数) - (两者同时充电次数) = T // r1 - T // lcm
      • shared_slots (共享时段数) = T - (无人机1充电次数) - (无人机2独占时段数) = T - (T // r1) - (T // r2 - T // lcm) = T - T // r1 - T // r2 + T // lcm

  2. 执行贪心分配

    • 无人机1需要完成 d1 次任务。它首先使用完自己的 slots_d1_only。还需要的任务数是 needed1 = max(0, d1 - slots_d1_only)。
    • 同理，无人机2在用完自己的 slots_d2_only 后，还需要的任务数是 needed2 = max(0, d2 - slots_d2_only)。

  3. 判定结果

    • 两者剩余所需要的任务 needed1 和 needed2 都必须从 shared_slots 中分配。
    • 如果共享时段足够满足两者的剩余需求，即 needed1 + needed2 <= shared_slots，那么时间 T 就是可行的，check(T) 返回 True。
    • 否则，返回 False。

复杂度分析

  • 时间复杂度: O(log K)，其中 K 是二分查找的搜索范围大小（约为 4 * 10¹⁴）。
    • 二分查找的迭代次数为 O(log K)。
    • 每次迭代中，check 函数只包含几次整数除法和乘法运算，以及一次 gcd 计算。gcd 的时间复杂度是 O(log(min(r1, r2)))。
    • 因此，总时间复杂度主要由二分查找的次数决定，执行效率非常高。
  • 空间复杂度: O(1)。
    • 除了存储输入和几个临时变量外，我们没有使用额外的空间。

总结

它巧妙地将一个复杂的“求解最优调度”问题，转化成了一个相对简单的、可以通过贪心策略解决的“可行性判定”问题。当我们遇到求解“最小值中的最大值”或“最大值中的最小值”这类问题，且问题的判定函数具有单调性时，二分查找往往是解决问题的黄金钥匙。它能帮助我们高效地逼近最优解，避免复杂的动态规划或搜索。
