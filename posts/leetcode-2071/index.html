LeetCode 2071 | 你可以安排的最多任务数目

发表于 2025-11-06 分类于 算法题解 ， 二分算法 ， 二分答案 ， 求最大 阅读次数： Valine：

LeetCode 第 2071 题：你可以安排的最多任务数目。
本题旨在求解“最多能完成的任务数”，这是一个典型的最优化问题。其核心在于识别出答案具有单调性，从而可以将求解问题转化为判定问题，利用二分查找在对数时间内高效求解。而其中的判定函数，则需要一个精巧的贪心策略来支撑。

问题解读

我们手上有 n 个任务和 m 个工人，每个任务有难度值 tasks[i]，每个工人有力量值 workers[j]。一个工人要完成一个任务，其力量值必须大于或等于任务的难度值。

此外，我们还有 pills 颗“神奇药丸”，每颗药丸可以给一个工人的力量值增加 strength。每个工人最多只能使用一颗药丸。

目标是找出在最优分配工人和药丸的情况下，最多可以完成多少个任务。

举个例子，tasks = [3,2,1], workers = [0,3,3], pills = 1, strength = 1。
我们可以尝试完成 3 个任务。

  • 最简单的任务是 1，最难的是 3。
  • 最弱的工人力量是 0，最强的是 3。
  • 一种可行的分配是：
    • 给力量为 0 的工人吃药，力量变为 0 + 1 = 1，刚好可以完成难度为 1 的任务。
    • 力量为 3 的工人可以完成难度为 2 的任务。
    • 另一个力量为 3 的工人可以完成难度为 3 的任务。
      这样，3 个任务都能完成。

这个例子暗示我们，为了完成尽可能多的任务，应该优先考虑让工人去完成那些他们力所能及的任务，并将宝贵的药丸用在刀刃上。

核心思路：将“求解问题”转化为“判定问题”与二分查找

直接求解“最多能完成多少个任务”是非常复杂的，因为它涉及到工人和任务的各种组合以及药丸的最佳分配策略。

但是，我们可以换一个角度思考：如果我们假定要完成 k 个任务，我们能否判断出这个目标是否可行？

这个问题就相对清晰了。我们可以设计一个辅助函数 check(k)，来判定“能否完成 k 个任务”。如果能解决这个判定问题，我们就能利用其单调性进行二分查找。

这个判定结果具有明显的单调性：

  • 如果我们能设法完成 k 个任务，那么我们必然也能完成 k-1 个任务（只需在完成 k 个任务的方案里去掉一个即可）。
  • 反之，如果我们连 k 个任务都无法完成，那么想完成 k+1 个任务就更不可能了。

这种单调性是应用二分查找的完美信号。我们可以对“能够完成的任务数量 k”进行二分查找，来快速定位到那个“最大可行”的临界点。

二分查找的搜索范围是什么呢？

  • 下界 (left)：0，最坏情况下我们一个任务也完不成。
  • 上界 (right)：min(n, m)，因为每个工人只能做一个任务，且任务总数和工人数是上限。

check(k) 的贪心策略

check(k) 函数是整个算法的核心。要判断能否完成 k 个任务，我们应该采用贪心策略来最大化成功的可能性：

  1. 排序：将 tasks 和 workers 数组都升序排序。
  2. 资源选择：要完成 k 个任务，最明智的选择是尝试完成 k 个最简单的任务，并为此派出 k 个最强大的工人。因为如果一个强大的工人都无法完成某个任务，那么更弱的工人就更不可能了。让强大的工人去应对挑战，可以为弱小的工人留下更简单的任务，从而最大化整体的成功率。
  3. 匹配与药丸分配：
    • 我们从这 k 个最强工人中的最弱者开始，依次为他们分配任务。
    • 对于一个工人 w，我们应该让他尝试完成当前所有未分配的 k 个最简单任务中，他力所能及的最简单的一个。这样做可以将更难的任务留给更强的工人。
    • 如果这个工人 w 连最简单的那个任务都无法独立完成（w < task_easiest），他就必须得用一颗药丸了。
    • 如果要用药丸，为了让药丸的价值最大化，应该让他去完成他能完成的最难的任务（w + strength >= task_hardest）。这样做，就把那些相对简单的任务，留给了后面那些可能不需要药丸就能完成任务的、更强的工人们。

这个策略可以用一个双端队列 (deque) 来完美实现。队列中存放 k 个最简单的任务。

  • 当工人 w 不需要药丸就能完成最简单的任务时，从队列头部取出一个任务 (deque.popleft())。
  • 当工人 w 必须使用药丸时，从队列尾部取出一个任务 (deque.pop())，并消耗一颗药丸。

算法详解

  1. 排序

    • 对 tasks 和 workers 数组进行升序排序。

  2. 确定二分查找的范围

    • l = 0：可能的最小完成数。
    • r = min(len(tasks), len(workers)) + 1：可能的最高完成数加一，作为开区间的右边界。

  3. 执行二分查找

    • 当 l + 1 < r 时，循环继续。
    • 计算中间值 mid = (l + r) // 2 作为当前的“猜测任务数”。
    • 调用辅助函数 check(mid) 来判定这个猜测值是否可行。
      • 如果 check(mid) 返回 True (即可以完成 mid 个任务)：
        • 说明 mid 是一个潜在的答案，我们也许还能完成更多。将搜索范围更新到右半边：l = mid。
      • 如果 check(mid) 返回 False (即无法完成 mid 个任务)：
        • 说明 mid 太大了，必须减少任务数。将搜索范围更新到左半边：r = mid。

  4. 返回结果

    • 循环结束后，l 中存储的就是满足条件的最多可完成任务数目。

复杂度分析

  • 时间复杂度: O(N log N + M log M + K log K)。
    • N 是任务数，M 是工人数。
    • K 是 min(N, M)。
    • 排序 tasks 和 workers 分别需要 O(N log N) 和 O(M log M)。
    • 二分查找在 [0, K] 范围内进行，需要 O(log K) 次迭代。
    • 在每次迭代中，check(k) 函数需要 O(k) 的时间来遍历工人和操作双端队列。
    • 因此，总时间复杂度为 O(N log N + M log M + K log K)。
  • 空间复杂度: O(K)。
    • 排序的空间复杂度取决于语言实现，可以是 O(log N + log M) 或 O(N + M)。
    • check 函数中使用的双端队列 valid_tasks 最多存储 k 个元素，因此需要 O(k) 的空间。

总结

本题是“二分答案”模型的又一个绝佳示例。当问题的目标是求解“最大值”或“最小值”，且答案满足单调性时，我们应优先考虑是否能将其转化为一个“判定性”问题。一旦转化成功，就可以套用二分查找的框架。

本题的难点在于设计 check(k) 函数的贪心策略。通过“排序 + 最优资源匹配（最强工人 vs 最简单任务）+ 双端队列”的组合，我们才能高效地判断出给定任务数 k 是否可行。这要求我们对贪心算法有深刻的理解，并能选择合适的数据结构来支撑策略的实现。
